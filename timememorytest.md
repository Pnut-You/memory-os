## 2. SQLite 是唯一事实来源

Time Memory 不从 Redis 快照、当前 Session 或滚动摘要中拼历史。生产链路为：

```text
每次对话
  → user / assistant 原文写入 SQLite events
  → 同一天可以产生多个 Session
  → 次日北京时间 01:00 触发日期抽取
  → 按 user_id + device_id + 当地自然日查询 SQLite
  → 读取该日全部 Session 的全部原始 message
  → 生成当天唯一一条 time_memory
  → 结果写回 SQLite events
```

所以，即使晚上还有活跃 Session，日期摘要也不能只取晚上的 Session。默认调度在日期结束后的次日 01:00 抽取前一自然日；手动重跑某一天也走相同 SQLite 日期查询。

Redis 只保存临时消息、快照和运行状态。Session 已关闭或 Redis TTL 到期后，SQLite 原始消息仍然会参与摘要。

正式评测复用了这条生产路径：每个 Case 的所有 Session 先写入临时 SQLite，再调用正式日期抽取入口，最后从 SQLite 读取生成的 `time_memory`。数据文件中的原始消息仅供 Judge 判卷核验，不会绕过 SQLite 直接交给摘要器。

自动化测试还覆盖了早上 07:00、下午 13:00、晚上 21:00 三个 Session。测试要求摘要器收到 SQLite 中全部 3 个 Session、6 条原始消息，并核对最终 Time Memory 的来源 Session ID 和事件 ID。

## 4. 人工事实标准

130 条正式 Case 包含 178 条人工事实：

| 每个 Case 的事实数 | Case 数 |
|---:|---:|
| 1 | 100 |
| 2 | 12 |
| 3 | 18 |

每条标准答案由完整事实和不可丢失字段组成，不再使用零散关键词：

```json
{
  "fact": "备用钥匙被放在玄关第二层抽屉里。",
  "critical_slots": {
    "object": "备用钥匙",
    "location": "玄关第二层抽屉"
  }
}
```

困难内容包括多事实、多数字、多人物、相似物品、位置细分、状态更新、否定、计划与完成、跨 Session 合并和普通闲聊干扰。

## 5. 具体如何判分

Judge 就是负责判卷的语言模型：

```text
模型：qwen3.5-flash-2026-02-23
服务：阿里云 DashScope OpenAI 兼容接口
API Key：优先 DASHSCOPE_API_KEY，否则 LLM_API_KEY
temperature：0
```

生成 Time Memory 和判卷默认使用同一个 Qwen 模型配置和同一个 API Key，但属于相互独立的请求。校准不会训练模型，也不会把标准答案展示给 Judge。

Judge 每次只判一条人工事实，同时得到：

1. SQLite 当天全部 Session 的完整原始对话；
2. 当前人工事实；
3. 当前事实的全部关键字段；
4. 生成的完整 Time Memory。

它不是检查关键词，也不是只比较两句话相不相似。它要先根据原始对话核验事实，再检查摘要中的人物、物品、位置、时间、数量、否定、完成状态和事实关系。

| 结果 | 含义 | 算正确 |
|---|---|---|
| 完全一致 | 完整事实和关键字段正确，基本没有改写 | 是 |
| 语义一致 | 合理改写，但全部事实和字段正确 | 是 |
| 部分保留 | 保留事件大意，但漏了至少一个关键字段 | 否 |
| 不匹配 | 事实遗漏、字段错误、状态反转、关系混淆或增加原文没有的事实 | 否 |

例如“十三度”变成“13℃”是语义一致；“备用钥匙放在玄关第二层抽屉”只写成“备用钥匙收好了”是部分保留。

程序还会检查 Judge 返回的槽位诊断是否自洽；有冲突或无依据扩写时不能保留正确标签。一个 Case 只有全部人工事实均为完全一致或语义一致才通过。

40 条人工固定校准样本中，四类各 10 条。它们只验证 Judge 是否可靠，不计入正式成绩。

|---|---:|---:|---:|---:|
| 完全一致 | 10 | 0 | 0 | 0 |
| 语义一致 | 0 | 10 | 0 | 0 |

|---|---:|---:|---:|

Judge 总准确率为 97.5%，高于 90% 安全门。唯一错误是一条人工“部分保留”被判成“语义一致”，说明 Judge 仍可能忽略某个精确关键字段。

## 7. 反事实验证

程序从正确摘要中选择 21 条，故意修改人物、位置、数量、时间、肯定/否定、完成状态，或删除一个关键字段，再交给同一个 Judge。

结果为 21 / 21 正确拒绝，拒绝率 100%，高于 95% 门槛。反事实门禁通过，说明 Judge 不是无条件给出正确标签。

## 8. Bad Case 总览

13 个 Case 未通过：

```text
time_004  time_048  time_054  time_080  time_103
time_107  time_108  time_116  time_117  time_118
time_119  time_126  time_128
```

这些失败可分成四类：

1. 真实摘要问题：无依据扩写、数字关系绑定、计划/完成或时间链混淆；
2. Judge 明显误判：摘要明确保留了关键字段，但 Judge 声称缺失；
3. 边界争议：事实本身保留，但整段摘要另有无依据扩写，导致所有 Fact 被统一降为不匹配；
4. API 超时：未完成判卷，不能证明摘要错误。

以下展示 6 个代表性案例。为便于阅读，省略“好、谢谢、明白”等纯填充轮次，但不省略事实相关消息。

### 案例 1：`time_048`——Judge 明显误判

原始事实消息：

```text
上午去图书馆归还了三本书。
```

人工事实：

```text
上午去图书馆归还了三本书。
```

完整生成摘要：

```text
日期总结（2026-05-18）
**2026-05-18 记忆摘要**

**今日主要事项：**
* **上午活动**：用户进行了桌面整理，随后归还了三本图书至图书馆。期间查看了天气并饮用了水。
* **中午安排**：短暂关闭了窗户。
* **下午工作**：处理了几封普通邮件。
* **晚间休息**：晚饭后在客厅休息。

**上下文与状态：**
* 全天对话中多次出现简短寒暄及确认性回复，未涉及新任务指令。
* 用户未提出任何待办事项、长期计划或需要后续跟进的请求。
* 无特殊环境变化记录或重要线索留存。

**未完成事项：**
* 无。
```

自动结果为不匹配。但“上午”“归还三本图书”“至图书馆”全部明确存在。这条更像 Judge 输出不完整后被程序保守归一为不匹配，不应当作 Time Memory 真实遗漏。

```text
Session 2：中午短暂关窗；下午处理普通邮件。
Session 3：晚饭后在客厅休息；晚上整理旅行拍摄的视频素材。
```

人工事实：

```text

```

完整生成摘要：

```text

**2026-05-24 记忆摘要**

**日常作息与环境**
用户今日生活节奏平稳。早晨进行了桌面简单整理，并饮水后查看了天气情况。中午曾短暂关闭窗户。晚间在客厅休息，整体居家状态放松。

**事务处理**
下午处理了数封普通邮件。晚间核心任务为整理旅行拍摄的视频素材，该事项已记录归档。

**上下文与线索**
1. **待办/进行中**：旅行视频素材的整理工作于晚间启动，需关注后续进度及完成状态。
```

### 案例 3：`time_080`——槽位语义解释错误

原始事实消息：

```text
今晚还要备份无人机拍摄文件。
```

人工事实：

```text
今晚还要备份无人机拍摄文件。
```

完整生成摘要：

```text
日期总结（2026-06-19）
**2026-06-19 时间记忆摘要**

**今日主要事项与行动：**
* **上午**：完成桌面整理；中午短暂关闭窗户。
* **下午**：处理数封普通邮件。
* **晚间**：晚饭后休息；睡前检查门窗。
* **待办/计划**：今晚需备份无人机拍摄文件（尚未完成，需重点关注）。

* **核心待办**：无人机拍摄文件的备份工作需在后续会话中提醒执行。
* **安全习惯**：已建立睡前检查门窗的固定流程，建议继续保持。
```

自动结果为部分保留。摘要第一处完整写出“今晚需备份无人机拍摄文件”。Judge 把“拍摄文件”误解成动作“拍摄”，再认为摘要只写了“备份”，属于槽位语义解释错误。摘要中的“固定安全习惯”和“建议保持”仍是无依据扩写，但目标事实没有遗漏。

## 9. 评测错误

`time_004`、`time_116` 和 `time_126` 各有一条事实未完成判卷，原因都是 Judge 连续两次在 75 秒读取超时：

| Case | 未完成判卷的人工事实 | 摘要是否明确包含 |
|---|---|---|
| `time_004` | 机器狗的新名字定为星尘 | 是，“新名字定为星尘” |
| `time_116` | 早上七点给十六盆花浇水 | 是，“07:00 完成十六盆花卉浇水” |
| `time_126` | 早上七点给九盆多肉浇水 | 是，“07:00 完成九盆多肉的浇水工作” |

这 3 条 Fact 没有被计成正确或不匹配，但仍保留在 178 条准确率分母中。控制台的 `Evaluation Errors: 3` 按 Case 计数；本轮恰好也是 3 条 Fact。

完整命令最终退出码为 4，原因是正式评测存在超时；Judge 校准门禁和反事实门禁均已通过。

## 10. 对结果的合理解释

相比关键词匹配，这套方法能接受“十三度/13℃”“晚上八点/20:00”“恢复正常功能/可正常使用”等合理改写，同时用关键字段阻止人物、位置、数量和状态被宽泛替代。

评测也不是只信任一个模型分数：

- 40 条人工标签先校准 Judge；
- 每条事实检查完整关键字段；
- 21 条故意改错摘要验证 Judge 会拒绝错误；
- 所有部分保留、不匹配、幻觉和超时写入人工审查文件；
- 控制台实时打印每条结果与原因。

但 Judge 本身仍会误判，特别是同义状态、槽位语义、跨 Session 相对时间，以及“整段摘要有一处幻觉时是否让该 Case 全部 Fact 失败”。因此：

> 85.96% 是可信的严格自动基线，可以暴露问题，但不能代替对失败 Case 的人工复核。当前摘要模型最明确的问题是喜欢增加原文没有的习惯、优先级、后续建议和事实关系；当前 Judge 最明确的问题是偶尔忽略摘要中已经存在的字段或对同义表达判得过严。
